  7  """Biopython interface for sequence search program outputs. 
  8   
  9  The SearchIO submodule provides parsers, indexers, and writers for outputs from 
 10  various sequence search programs. It provides an API similar to SeqIO and 
 11  AlignIO, with the following main functions: ``parse``, ``read``, ``to_dict``, 
 12  ``index``, ``index_db``, ``write``, and ``convert``. 
 13   
 14  SearchIO parses a search output file's contents into a hierarchy of four nested 
 15  objects: QueryResult, Hit, HSP, and HSPFragment. Each of them models a part of 
 16  the search output file: 
 17   
 18      - QueryResult represents a search query. This is the main object returned 
 19        by the input functions and it contains all other objects. 
 20      - Hit represents a database hit, 
 21      - HSP represents high-scoring alignment region(s) in the hit, 
 22      - HSPFragment represents a contiguous alignment within the HSP 
 23   
 24  In addition to the four objects above, SearchIO is also tightly integrated with 
 25  the SeqRecord objects (see SeqIO) and MultipleSeqAlignment objects (see 
 26  AlignIO). SeqRecord objects are used to store the actual matching hit and query 
 27  sequences, while MultipleSeqAlignment objects stores the alignment between them. 
 28   
 29  A detailed description of these objects' features and their example usages are 
 30  available in their respective documentations. 
 31   
 32   
 33  Input 
 34  ===== 
 35  The main function for parsing search output files is Bio.SearchIO.parse(...). 
 36  This function parses a given search output file and returns a generator object 
 37  that yields one QueryResult object per iteration. 
 38   
 39  ``parse`` takes two arguments: 1) a file handle or a filename of the input file 
 40  (the search output file) and 2) the format name. 
 41   
 42      >>> from Bio import SearchIO 
 43      >>> for qresult in SearchIO.parse('Blast/mirna.xml', 'blast-xml'): 
 44      ...     print("%s %s" % (qresult.id, qresult.description)) 
 45      ... 
 46      33211 mir_1 
 47      33212 mir_2 
 48      33213 mir_3 
 49   
 50  SearchIO also provides the Bio.SearchIO.read(...) function, which is intended 
 51  for use on search output files containing only one query. ``read`` returns one 
 52  QueryResult object and will raise an exception if the source file contains more 
 53  than one queries: 
 54   
 55      >>> qresult = SearchIO.read('Blast/xml_2226_blastp_004.xml', 'blast-xml') 
 56      >>> print("%s %s" % (qresult.id, qresult.description)) 
 57      ... 
 58      gi|11464971:4-101 pleckstrin [Mus musculus] 
 59   
 60      >>> SearchIO.read('Blast/mirna.xml', 'blast-xml') 
 61      Traceback (most recent call last): 
 62      ... 
 63      ValueError: ... 
 64   
 65  For accessing search results of large output files, you may use the indexing 
 66  functions Bio.SearchIO.index(...) or Bio.SearchIO.index_db(...). They have a 
 67  similar interface to their counterparts in SeqIO and AlignIO, with the addition 
 68  of optional, format-specific keyword arguments. 
 69   
 70   
 71  Output 
 72  ====== 
 73  SearchIO has writing support for several formats, accessible from the 
 74  Bio.SearchIO.write(...) function. This function returns a tuple of four 
 75  numbers: the number of QueryResult, Hit, HSP, and HSPFragment written:: 
 76   
 77      qresults = SearchIO.parse('Blast/mirna.xml', 'blast-xml') 
 78      SearchIO.write(qresults, 'results.tab', 'blast-tab') 
 79      <stdout> (3, 239, 277, 277) 
 80   
 81  Note that different writers may require different attribute values of the 
 82  SearchIO objects. This limits the scope of writable search results to search 
 83  results possessing the required attributes. 
 84   
 85  For example, the writer for HMMER domain table output requires 
 86  the conditional e-value attribute from each HSP object, among others. If you 
 87  try to write to the HMMER domain table format and your HSPs do not have this 
 88  attribute, an exception will be raised. 
 89   
 90   
 91  Conversion 
 92  ========== 
 93  SearchIO provides a shortcut function Bio.SearchIO.convert(...) to convert a 
 94  given file into another format. Under the hood, ``convert`` simply parses a given 
 95  output file and writes it to another using the ``parse`` and ``write`` functions. 
 96   
 97  Note that the same restrictions found in Bio.SearchIO.write(...) applies to the 
 98  convert function as well. 
 99   
100   
101  Conventions 
102  =========== 
103  The main goal of creating SearchIO is to have a common, easy to use interface 
104  across different search output files. As such, we have also created some 
105  conventions / standards for SearchIO that extend beyond the common object model. 
106  These conventions apply to all files parsed by SearchIO, regardless of their 
107  individual formats. 
108   
109  Python-style sequence coordinates 
110  --------------------------------- 
111   
112  When storing sequence coordinates (start and end values), SearchIO uses 
113  the Python-style slice convention: zero-based and half-open intervals. For 
114  example, if in a BLAST XML output file the start and end coordinates of an 
115  HSP are 10 and 28, they would become 9 and 28 in SearchIO. The start 
116  coordinate becomes 9 because Python indices start from zero, while the end 
117  coordinate remains 28 as Python slices omit the last item in an interval. 
118   
119  Beside giving you the benefits of standardization, this convention also 
120  makes the coordinates usable for slicing sequences. For example, given a 
121  full query sequence and the start and end coordinates of an HSP, one can 
122  use the coordinates to extract part of the query sequence that results in 
123  the database hit. 
124   
125  When these objects are written to an output file using 
126  SearchIO.write(...), the coordinate values are restored to their 
127  respective format's convention. Using the example above, if the HSP would 
128  be written to an XML file, the start and end coordinates would become 10 
129  and 28 again. 
130   
131  Sequence coordinate order 
132  ------------------------- 
133   
134  Some search output format reverses the start and end coordinate sequences 
135  according to the sequence's strand. For example, in BLAST plain text 
136  format if the matching strand lies in the minus orientation, then the 
137  start coordinate will always be bigger than the end coordinate. 
138   
139  In SearchIO, start coordinates are always smaller than the end 
140  coordinates, regardless of their originating strand. This ensures 
141  consistency when using the coordinates to slice full sequences. 
142   
143  Note that this coordinate order convention is only enforced in the 
144  HSPFragment level. If an HSP object has several HSPFragment objects, each 
145  individual fragment will conform to this convention. But the order of the 
146  fragments within the HSP object follows what the search output file uses. 
147   
148  Similar to the coordinate style convention, the start and end coordinates' 
149  order are restored to their respective formats when the objects are 
150  written using Bio.SearchIO.write(...). 
151   
152  Frames and strand values 
153  ------------------------ 
154   
155  SearchIO only allows -1, 0, 1 and None as strand values. For frames, the 
156  only allowed values are integers from -3 to 3 (inclusive) and None. Both 
157  of these are standard Biopython conventions. 
158   
159   
160  Supported Formats 
161  ================= 
162  Below is a list of search program output formats supported by SearchIO. 
163   
164  Support for parsing, indexing, and writing: 
165   
166   - blast-tab        - BLAST+ tabular output. Both variants without comments 
167                        (-m 6 flag) and with comments (-m 7 flag) are supported. 
168   - blast-xml        - BLAST+ XML output. 
169   - blat-psl         - The default output of BLAT (PSL format). Variants with or 
170                        without header are both supported. PSLX (PSL + sequences) 
171                        is also supported. 
172   - hmmer3-tab       - HMMER3 table output. 
173   - hmmer3-domtab    - HMMER3 domain table output. When using this format, the 
174                        program name has to be specified. For example, for parsing 
175                        hmmscan output, the name would be 'hmmscan-domtab'. 
176   
177  Support for parsing and indexing: 
178   
179   - exonerate-text   - Exonerate plain text output. 
180   - exonerate-vulgar - Exonerate vulgar line. 
181   - exonerate-cigar  - Exonerate cigar line. 
182   - fasta-m10        - Bill Pearson's FASTA -m 10 output. 
183   - hmmer3-text      - HMMER3 regular text output format. Supported HMMER3 
184                        subprograms are hmmscan, hmmsearch, and phmmer. 
185   - hmmer2-text      - HMMER2 regular text output format. Supported HMMER2 
186                        subprograms are hmmpfam, hmmsearch. 
187   
188  Support for parsing: 
189   
190   - blast-text       - BLAST+ plain text output. 
191   - hhsuite2-text    - HHSUITE plain text output. 
192   
193  Each of these formats have different keyword arguments available for use with 
194  the main SearchIO functions. More details and examples are available in each 
195  of the format's documentation.
