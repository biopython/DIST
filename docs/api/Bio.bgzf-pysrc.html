  8   
  9  The SAM/BAM file format (Sequence Alignment/Map) comes in a plain text 
 10  format (SAM), and a compressed binary format (BAM). The latter uses a 
 11  modified form of gzip compression called BGZF (Blocked GNU Zip Format), 
 12  which can be applied to any file format to provide compression with 
 13  efficient random access. BGZF is described together with the SAM/BAM 
 14  file format at http://samtools.sourceforge.net/SAM1.pdf 
 15   
 16  Please read the text below about 'virtual offsets' before using BGZF 
 17  files for random access. 
 18   
 19  Aim of this module 
 20  ------------------ 
 21  The Python gzip library can be used to read BGZF files, since for 
 22  decompression they are just (specialised) gzip files. What this 
 23  module aims to facilitate is random access to BGZF files (using the 
 24  'virtual offset' idea), and writing BGZF files (which means using 
 25  suitably sized gzip blocks and writing the extra 'BC' field in the 
 26  gzip headers). As in the gzip library, the zlib library is used 
 27  internally. 
 28   
 29  In addition to being required for random access to and writing of 
 30  BAM files, the BGZF format can also be used on other sequential 
 31  data (in the sense of one record after another), such as most of 
 32  the sequence data formats supported in Bio.SeqIO (like FASTA, 
 33  FASTQ, GenBank, etc) or large MAF alignments. 
 34   
 35  The Bio.SeqIO indexing functions use this module to support BGZF files. 
 36   
 37  Technical Introduction to BGZF 
 38  ------------------------------ 
 39  The gzip file format allows multiple compressed blocks, each of which 
 40  could be a stand alone gzip file. As an interesting bonus, this means 
 41  you can use Unix ``cat`` to combine two or more gzip files into one by 
 42  concatenating them. Also, each block can have one of several compression 
 43  levels (including uncompressed, which actually takes up a little bit 
 44  more space due to the gzip header). 
 45   
 46  What the BAM designers realised was that while random access to data 
 47  stored in traditional gzip files was slow, breaking the file into 
 48  gzip blocks would allow fast random access to each block. To access 
 49  a particular piece of the decompressed data, you just need to know 
 50  which block it starts in (the offset of the gzip block start), and 
 51  how far into the (decompressed) contents of the block you need to 
 52  read. 
 53   
 54  One problem with this is finding the gzip block sizes efficiently. 
 55  You can do it with a standard gzip file, but it requires every block 
 56  to be decompressed -- and that would be rather slow. Additionally 
 57  typical gzip files may use very large blocks. 
 58   
 59  All that differs in BGZF is that compressed size of each gzip block 
 60  is limited to 2^16 bytes, and an extra 'BC' field in the gzip header 
 61  records this size. Traditional decompression tools can ignore this, 
 62  and unzip the file just like any other gzip file. 
 63   
 64  The point of this is you can look at the first BGZF block, find out 
 65  how big it is from this 'BC' header, and thus seek immediately to 
 66  the second block, and so on. 
 67   
 68  The BAM indexing scheme records read positions using a 64 bit 
 69  'virtual offset', comprising ``coffset << 16 | uoffset``, where ``coffset`` 
 70  is the file offset of the BGZF block containing the start of the read 
 71  (unsigned integer using up to 64-16 = 48 bits), and ``uoffset`` is the 
 72  offset within the (decompressed) block (unsigned 16 bit integer). 
 73   
 74  This limits you to BAM files where the last block starts by 2^48 
 75  bytes, or 256 petabytes, and the decompressed size of each block 
 76  is at most 2^16 bytes, or 64kb. Note that this matches the BGZF 
 77  'BC' field size which limits the compressed size of each block to 
 78  2^16 bytes, allowing for BAM files to use BGZF with no gzip 
 79  compression (useful for intermediate files in memory to reduce 
 80  CPU load). 
 81   
 82  Warning about namespaces 
 83  ------------------------ 
 84  It is considered a bad idea to use "from XXX import ``*``" in Python, because 
 85  it pollutes the namespace. This is a real issue with Bio.bgzf (and the 
 86  standard Python library gzip) because they contain a function called open 
 87  i.e. Suppose you do this: 
 88   
 89  >>> from Bio.bgzf import * 
 90  >>> print(open.__module__) 
 91  Bio.bgzf 
 92   
 93  Or,

 95  >>> from gzip import * 
 96  >>> print(open.__module__) 
 97  gzip 
 98

100  need to by importing the built-in open function: 
101

103  ...     from __builtin__ import open # Python 2 
104  ... except ImportError: 
105  ...     from builtins import open # Python 3 
106  ...

108  However, what we recommend instead is to use the explicit namespace, e.g. 
109

164  Raw start 92243, raw length 28; data start 305622, data length 0 
165  >>> handle.close()

167  In this example the first three blocks are 'full' and hold 65536 bytes 
168  of uncompressed data. The fourth block isn't full and holds 43478 bytes. 
169  Finally there is a special empty fifth block which takes 28 bytes on 
170  disk and serves as an 'end of file' (EOF) marker. If this is missing, 
171  it is possible your BGZF file is incomplete. 
172   
173  By reading ahead 70,000 bytes we moved into the second BGZF block, 
174  and at that point the BGZF virtual offsets start to look different 
175  to a simple offset into the decompressed data as exposed by the gzip 
176  library. 
177   
178  As an example, consider seeking to the decompressed position 196734. 
179  Since 196734 = 65536 + 65536 + 65536 + 126 = 65536*3 + 126, this 
180  is equivalent to jumping the first three blocks (which in this 
181  specific example are all size 65536 after decompression - which 
182  does not always hold) and starting at byte 126 of the fourth block 
183  (after decompression). For BGZF, we need to know the fourth block's 
184  offset of 55074 and the offset within the block of 126 to get the 
185  BGZF virtual offset. 
186   
187  >>> print(55074 << 16 | 126) 
188  3609329790 
189  >>> print(bgzf.make_virtual_offset(55074, 126)) 
190  3609329790 
191   
192  Thus for this BGZF file, decompressed position 196734 corresponds 
193  to the virtual offset 3609329790. However, another BGZF file with 
194  different contents would have compressed more or less efficiently, 
195  so the compressed blocks would be different sizes. What this means 
196  is the mapping between the uncompressed offset and the compressed 
197  virtual offset depends on the BGZF file you are using. 
198   
199  If you are accessing a BGZF file via this module, just use the 
200  handle.tell() method to note the virtual offset of a position you 
201  may later want to return to using handle.seek(). 
202   
203  The catch with BGZF virtual offsets is while they can be compared 
204  (which offset comes first in the file), you cannot safely subtract 
205  them to get the size of the data between them, nor add/subtract 
206  a relative offset. 
207   
208  Of course you can parse this file with Bio.SeqIO using BgzfReader, 
209  although there isn't any benefit over using gzip.open(...), unless 
210  you want to index BGZF compressed sequence files: 
211   
212  >>> from Bio import SeqIO 
213  >>> handle = BgzfReader("GenBank/NC_000932.gb.bgz") 
214  >>> record = SeqIO.read(handle, "genbank")
