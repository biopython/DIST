  21  Input 
  22  ----- 
  23  The main function is Bio.SeqIO.parse(...) which takes an input file handle 
  24  (or in recent versions of Biopython alternatively a filename as a string), 
  25  and format string.  This returns an iterator giving SeqRecord objects: 
  26   
  27  >>> from Bio import SeqIO 
  28  >>> for record in SeqIO.parse("Fasta/f002", "fasta"): 
  29  ...     print("%s %i" % (record.id, len(record))) 
  30  gi|1348912|gb|G26680|G26680 633 
  31  gi|1348917|gb|G26685|G26685 413 
  32  gi|1592936|gb|G29385|G29385 471 
  33   
  34  Note that the parse() function will invoke the relevant parser for the 
  35  format with its default settings.  You may want more control, in which case 
  36  you need to create a format specific sequence iterator directly. 
  37   
  38  Some of these parsers are wrappers around low-level parsers which build up 
  39  SeqRecord objects for the consistent SeqIO interface. In cases where the 
  40  run-time is critical, such as large FASTA or FASTQ files, calling these 
  41  underlying parsers will be much faster - in this case these generator 
  42  functions which return tuples of strings: 
  43   
  44  >>> from Bio.SeqIO.FastaIO import SimpleFastaParser 
  45  >>> from Bio.SeqIO.QualityIO import FastqGeneralIterator 
  46   
  47   
  48  Input - Single Records 
  49  ---------------------- 
  50  If you expect your file to contain one-and-only-one record, then we provide 
  51  the following 'helper' function which will return a single SeqRecord, or 
  52  raise an exception if there are no records or more than one record: 
  53   
  54  >>> from Bio import SeqIO 
  55  >>> record = SeqIO.read("Fasta/f001", "fasta") 
  56  >>> print("%s %i" % (record.id, len(record))) 
  57  gi|3318709|pdb|1A91| 79 
  58   
  59  This style is useful when you expect a single record only (and would 
  60  consider multiple records an error).  For example, when dealing with GenBank 
  61  files for bacterial genomes or chromosomes, there is normally only a single 
  62  record.  Alternatively, use this with a handle when downloading a single 
  63  record from the internet. 
  64   
  65  However, if you just want the first record from a file containing multiple 
  66  record, use the next() function on the iterator (or under Python 2, the 
  67  iterator's next() method): 
  68   
  69  >>> from Bio import SeqIO 
  70  >>> record = next(SeqIO.parse("Fasta/f002", "fasta")) 
  71  >>> print("%s %i" % (record.id, len(record))) 
  72  gi|1348912|gb|G26680|G26680 633 
  73   
  74  The above code will work as long as the file contains at least one record. 
  75  Note that if there is more than one record, the remaining records will be 
  76  silently ignored. 
  77   
  78   
  79  Input - Multiple Records 
  80  ------------------------ 
  81  For non-interlaced files (e.g. Fasta, GenBank, EMBL) with multiple records 
  82  using a sequence iterator can save you a lot of memory (RAM).  There is 
  83  less benefit for interlaced file formats (e.g. most multiple alignment file 
  84  formats).  However, an iterator only lets you access the records one by one. 
  85   
  86  If you want random access to the records by number, turn this into a list: 
  87   
  88  >>> from Bio import SeqIO 
  89  >>> records = list(SeqIO.parse("Fasta/f002", "fasta")) 
  90  >>> len(records) 
  91  3 
  92  >>> print(records[1].id) 
  93  gi|1348917|gb|G26685|G26685 
  94   
  95  If you want random access to the records by a key such as the record id, 
  96  turn the iterator into a dictionary: 
  97   
  98  >>> from Bio import SeqIO 
  99  >>> record_dict = SeqIO.to_dict(SeqIO.parse("Fasta/f002", "fasta")) 
 100  >>> len(record_dict) 
 101  3 
 102  >>> print(len(record_dict["gi|1348917|gb|G26685|G26685"])) 
 103  413 
 104   
 105  However, using list() or the to_dict() function will load all the records 
 106  into memory at once, and therefore is not possible on very large files. 
 107  Instead, for *some* file formats Bio.SeqIO provides an indexing approach 
 108  providing dictionary like access to any record. For example, 
 109   
 110  >>> from Bio import SeqIO 
 111  >>> record_dict = SeqIO.index("Fasta/f002", "fasta") 
 112  >>> len(record_dict) 
 113  3 
 114  >>> print(len(record_dict["gi|1348917|gb|G26685|G26685"])) 
 115  413 
 116  >>> record_dict.close() 
 117   
 118  Many but not all of the supported input file formats can be indexed like 
 119  this. For example "fasta", "fastq", "qual" and even the binary format "sff" 
 120  work, but alignment formats like "phylip", "clustalw" and "nexus" will not. 
 121   
 122  In most cases you can also use SeqIO.index to get the record from the file 
 123  as a raw string (not a SeqRecord). This can be useful for example to extract 
 124  a sub-set of records from a file where SeqIO cannot output the file format 
 125  (e.g. the plain text SwissProt format, "swiss") or where it is important to 
 126  keep the output 100% identical to the input). For example, 
 127   
 128  >>> from Bio import SeqIO 
 129  >>> record_dict = SeqIO.index("Fasta/f002", "fasta") 
 130  >>> len(record_dict) 
 131  3 
 132  >>> print(record_dict.get_raw("gi|1348917|gb|G26685|G26685").decode())

 139  AATTGNTTTGCCCTTTTGGGTTTNTTNTTGGTAAANNNTTCCCGGGTGGGGGNGGTNNNGAAA 
 140  <BLANKLINE>

 142  >gi|1348917|gb|G26685|G26685 human STS STS_D11734. 
 143  CGGAGCCAGCGAGCATATGCTGCATGAGGACCTTTCTATCTTACATTATGGCTGGGAATC

 197  Input - Alignments 
 198  ------------------ 
 199  You can read in alignment files as alignment objects using Bio.AlignIO. 
 200  Alternatively, reading in an alignment file format via Bio.SeqIO will give 
 201  you a SeqRecord for each row of each alignment: 
 202   
 203  >>> from Bio import SeqIO 
 204  >>> for record in SeqIO.parse("Clustalw/hedgehog.aln", "clustal"): 
 205  ...     print("%s %i" % (record.id, len(record))) 
 206  gi|167877390|gb|EDS40773.1| 447 
 207  gi|167234445|ref|NP_001107837. 447 
 208  gi|74100009|gb|AAZ99217.1| 447 
 209  gi|13990994|dbj|BAA33523.2| 447 
 210  gi|56122354|gb|AAV74328.1| 447

 213  Output 
 214  ------ 
 215  Use the function Bio.SeqIO.write(...), which takes a complete set of 
 216  SeqRecord objects (either as a list, or an iterator), an output file handle 
 217  (or in recent versions of Biopython an output filename as a string) and of 
 218  course the file format:: 
 219   
 220    from Bio import SeqIO 
 221    records = ... 
 222    SeqIO.write(records, "example.faa", "fasta") 
 223   
 224  Or, using a handle:: 
 225   
 226      from Bio import SeqIO 
 227      records = ... 
 228      with open("example.faa", "w") as handle: 
 229        SeqIO.write(records, handle, "fasta") 
 230   
 231  You are expected to call this function once (with all your records) and if 
 232  using a handle, make sure you close it to flush the data to the hard disk. 
 233   
 234   
 235  Output - Advanced 
 236  ----------------- 
 237  The effect of calling write() multiple times on a single file will vary 
 238  depending on the file format, and is best avoided unless you have a strong 
 239  reason to do so. 
 240   
 241  If you give a filename, then each time you call write() the existing file 
 242  will be overwritten. For sequential files formats (e.g. fasta, genbank) each 
 243  "record block" holds a single sequence.  For these files it would probably 
 244  be safe to call write() multiple times by re-using the same handle. 
 245   
 246  However, trying this for certain alignment formats (e.g. phylip, clustal, 
 247  stockholm) would have the effect of concatenating several multiple sequence 
 248  alignments together.  Such files are created by the PHYLIP suite of programs 
 249  for bootstrap analysis, but it is clearer to do this via Bio.AlignIO instead. 
 250   
 251  Worse, many fileformats have an explicit header and/or footer structure 
 252  (e.g. any XMl format, and most binary file formats like SFF). Here making 
 253  multiple calls to write() will result in an invalid file. 
 254   
 255   
 256  Conversion 
 257  ---------- 
 258  The Bio.SeqIO.convert(...) function allows an easy interface for simple 
 259  file format conversions. Additionally, it may use file format specific 
 260  optimisations so this should be the fastest way too. 
 261   
 262  In general however, you can combine the Bio.SeqIO.parse(...) function with 
 263  the Bio.SeqIO.write(...) function for sequence file conversion. Using 
 264  generator expressions or generator functions provides a memory efficient way 
 265  to perform filtering or other extra operations as part of the process. 
 266   
 267   
 268  File Formats 
 269  ------------ 
 270  When specifying the file format, use lowercase strings.  The same format 
 271  names are also used in Bio.AlignIO and include the following: 
 272   
 273      - abi     - Applied Biosystem's sequencing trace format 
 274      - abi-trim - Same as "abi" but with quality trimming with Mott's algorithm 
 275      - ace     - Reads the contig sequences from an ACE assembly file. 
 276      - cif-atom - Uses Bio.PDB.MMCIFParser to determine the (partial) protein 
 277        sequence as it appears in the structure based on the atomic coordinates. 
 278      - cif-seqres - Reads a macromolecular Crystallographic Information File 
 279        (mmCIF) file to determine the complete protein sequence as defined by the 
 280        _pdbx_poly_seq_scheme records. 
 281      - embl    - The EMBL flat file format. Uses Bio.GenBank internally. 
 282      - fasta   - The generic sequence file format where each record starts with 
 283        an identifer line starting with a ">" character, followed by 
 284        lines of sequence. 
 285      - fasta-2line - Stricter interpretation of the FASTA format using exactly 
 286        two lines per record (no line wrapping). 
 287      - fastq   - A "FASTA like" format used by Sanger which also stores PHRED 
 288        sequence quality values (with an ASCII offset of 33). 
 289      - fastq-sanger - An alias for "fastq" for consistency with BioPerl and EMBOSS 
 290      - fastq-solexa - Original Solexa/Illumnia variant of the FASTQ format which 
 291        encodes Solexa quality scores (not PHRED quality scores) with an 
 292        ASCII offset of 64. 
 293      - fastq-illumina - Solexa/Illumina 1.3 to 1.7 variant of the FASTQ format 
 294        which encodes PHRED quality scores with an ASCII offset of 64 
 295        (not 33). Note as of version 1.8 of the CASAVA pipeline Illumina 
 296        will produce FASTQ files using the standard Sanger encoding. 
 297      - genbank - The GenBank or GenPept flat file format. 
 298      - gb      - An alias for "genbank", for consistency with NCBI Entrez Utilities 
 299      - ig      - The IntelliGenetics file format, apparently the same as the 
 300        MASE alignment format. 
 301      - imgt    - An EMBL like format from IMGT where the feature tables are more 
 302        indented to allow for longer feature types. 
 303      - pdb-seqres -  Reads a Protein Data Bank (PDB) file to determine the 
 304        complete protein sequence as it appears in the header (no dependencies). 
 305      - pdb-atom - Uses Bio.PDB to determine the (partial) protein sequence as 
 306        it appears in the structure based on the atom coordinate section of the 
 307        file (requires NumPy for Bio.PDB). 
 308      - phd     - Output from PHRED, used by PHRAP and CONSED for input. 
 309      - pir     - A "FASTA like" format introduced by the National Biomedical 
 310        Research Foundation (NBRF) for the Protein Information Resource 
 311        (PIR) database, now part of UniProt. 
 312      - seqxml  - SeqXML, simple XML format described in Schmitt et al (2011). 
 313      - sff     - Standard Flowgram Format (SFF), typical output from Roche 454. 
 314      - sff-trim - Standard Flowgram Format (SFF) with given trimming applied. 
 315      - swiss   - Plain text Swiss-Prot aka UniProt format. 
 316      - tab     - Simple two column tab separated sequence files, where each 
 317        line holds a record's identifier and sequence. For example, 
 318        this is used as by Aligent's eArray software when saving 
 319        microarray probes in a minimal tab delimited text file. 
 320      - qual    - A "FASTA like" format holding PHRED quality values from 
 321        sequencing DNA, but no actual sequences (usually provided 
 322        in separate FASTA files). 
 323      - uniprot-xml - The UniProt XML format (replacement for the SwissProt plain 
 324        text format which we call "swiss") 
 325   
 326  Note that while Bio.SeqIO can read all the above file formats, it cannot 
 327  write to all of them. 
 328   
 329  You can also use any file format supported by Bio.AlignIO, such as "nexus", 
 330  "phylip" and "stockholm", which gives you access to the individual sequences 
 331  making up each alignment as SeqRecords.
